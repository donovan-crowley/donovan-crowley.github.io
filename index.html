Electric Vehicle

Our team won 3rd place in the Novice Division at the 2023 Lime Rock Park Electrathon Competition

Background

We built an electric vehicle from scratch for an endurance competition, developing a car that could go the most distance in an hour.

Important Components

  • Two 12V Yellow Top Optima lead-acid batteries, providing a total voltage of 24V for high specific energy
  • Alltrax Motor Controller
  • DC Motor with 2800 RPM
  • Bicycle Disc Brakes

    • Tools Used

  • Fusion 360 & 3D Printer to design and print parts that protected the electrical components and connected awkward pieces such as the mirrors and fan
  • Soldering iron to connect electronics
  • Hot glue gun, string, and tape to connect parts to the chassis
  • Digital multimeter to measure voltage, current, and resistance at different parts of the circuit

    • Process

    While originally tasked with developing the foundations of the vehicle including the chassis, mirrors, throttle, brakes, steering wheel, and chair, I ran into problems with the electrical system.

    Seeing this challenge would eventually need to be overcome, I spearheaded the development of the circuit design. The given schematic for the motor controller appeared to be incorrect, so I researched, tested, and implemented my own adaptation inspired by the original to power the vehicle.


    Circuit

    The simplest loop in the circuit was connecting the controller pins to the potentiometer that controls the throttle as shown in the original schematic. The rest of the circuit was not as easy, and we could never produce power to rotate the motor sprocket following the original.

    So, I continuously redesigned and tested the schematic until I found my final design, as shown above, that incorporated a brake light and display monitor. Once the vehicle was able to run, we focused on perfecting the other aspects.

    Calculations

    For the endurance aspect of the competition, we looked to maximize our performance by analyzing the weight and energy cost of the vehicle.

    The competition required that the minimum weight of the driver with additional weight to the vehicle must be 80kg. Since both drivers were under 80kg, we stacked our motor, batteries, and all the electronics in the back and added our ballast to the front. This distribution proved advantageous, allowing the vehicle to better climb the hills in the course.

    We stayed around 30-40mph throughout the entire race to conserve energy and maintain a relatively fast speed. We used more energy to climb hills and on the straight-aways than while turning or descending.

    I measured the center-to-center distance between the motor and wheel to be about 16", ensuring little slack with the gear chain. Through online research, I determined a 6:1 motor-to-sprocket ratio to be most efficient for the endurance, turns, and hills of the competition.

    Web Scraper

    I built a web scraper using Puppeteer from the Node.js library, which provides an API for controlling headless Chrome and Chrome browsers and Cheerio, which has jQuery-like features, to parse HTML and manimulate DOM on the server side.

    Currently, the web scraper extracts basic information such as weather and a daily quote, and I looked to develop a more complex implementation.

    The Wiki Game contains a popular problem that searches for the shortest path between pages on Wikipedia using internal links. My initial implementation included the Puppeteer and Cheerio installations, but the process of opening, reading, and closing browser instances was slow and inefficient.

    Thus, I transitioned to use Wikipedia's public API with Axios to fetch page data directly and extrapolate internal links using a breadth-first search approach. The process still takes a considerable amount of time, yet it is much faster than my initial approach. Also, it's important to ensure the start and end links exist before the code fetches multiple Wikipedia pages, skipping the desired end.

    Then, I switched to Python to implement the SentenceTransformer pretrained language model. Additionally, I switched to an A* search to embed each link and use cosine similarity to compare the links based on the given end goal with the SentenceTransformers library. A higher cosine similarity meant the link was was semantically closer to the goal and that link would be pushed to the top of a priority-queue to be analyzed further.

    Object Tracker


    Overview

    Developed a real-time object tracking algorithm in C++ using OpenCV. The program captures live video from a webcam using OpenCV's VideoCapture, detects circular objects, learns the object's color characteristics, and tracks the object frame by frame.

    SEARCH

    The default search mode converts the frames to grayscale and detects objects using the Hough Circle Transform.

    TRAIN

    Once a target is found, the train mode is deployed by typing the key "t" on the computer. The algorithm samples pixels inside and outside the circle to calculate the average HSV (Hue, Saturation, Value) values for color boundaries. The program iteratively adjusts the thresholds to maximize the difference between the internal and external color values of the circle. When the optimal threshold is found, the program automatically transitions into the tracking mode.

    TRACK

    Using the learned HSV range to create a binary mask, the tracking mode applies morphological operations to reduce noise and focus on the object to track. The programs tracks the center of the object using image moments and traces a line to display the objects movement.

    READ

    I used the read mode more for the development of the project to familiarize myself with OpenCV HSV values. This mode calculates the hue, saturation, and value within a particular fixed region to display real-time color values.
